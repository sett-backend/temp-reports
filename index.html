Key Patterns Worth Investigating
🔁
Compliance issues keep recurring across customer accounts
22 tasks (34%) had compliance-related revisions — misleading earnings claims, missing disclaimers, and Sett branding still appearing in shipped builds. These are the same types of issues appearing across free-cash, bubble-cash, solitaire-cash, and others. The Sett logo appearing in 6 tasks suggests it may be a default template issue that Apollo doesn't know to remove. The "you can win X dollars" vs "you could have won" distinction appears in at least 9 tasks, suggesting auditors are having to manually catch something that could be a templated check.
Task 2024 (solitaire-cash): "let's avoid saying players will win a specific amount, since we can't actually guarantee that" — same feedback seen on tasks 1012, 689, 2266, 2563, 1778, 2567, 1125
📦
Missing network builds is a tail-end delivery problem
21 tasks (32%) were flagged for missing network exports — Mintegral, Liftoff, Moloco, Facebook, Smadex, Snapchat. This happens consistently at the end of the delivery process after the PG reviews the build. Apollo likely completes the creative work but doesn't know which networks the customer needs. The customer specification isn't being captured upfront, creating a predictable re-work cycle at the end of every delivery.
Task 3330 (free-cash): "mintegral and liftoff builds are missing" — then "Also add Snapchat network build" — two separate revision rounds for the same issue
🎮
Game mechanic / experience issues are the most common category but hardest to catch early
55% of tasks had mechanic-related revisions — wrong feel, wrong pacing, experience that "doesn't create desire to play." These are harder to systematize than compliance or export issues. The clearest pattern: Apollo delivers builds that are technically correct but experientially flat. The Solitaire Grand Harvest example (3 failed taps before any successful card flip) and Woodoku Blast (concept rework needed) suggest Apollo may be optimizing for spec-compliance rather than experience quality.
Task 2973 (solitaire-grand-harvest): "The experience still feels like a fail state. Currently there are 3 taps that don't flip any card, and on the 4th tap the user is sent to the store — this doesn't create any desire to play. The player must see successful card flips from the very beginning."
🔄
High repeat-message rate suggests fixes aren't sticking
207 feedback messages across 65 tasks = 3.2 average. But some tasks have far more: Task 2124 (match-factory) had 7 messages about the same 60s timer bug. Task 3179 (bubble-cash) had 3 messages about "CTA out of screen" before it was fixed. Task 2595 (bubble-cash) had 5 identical messages about luxury items claim. When auditors have to repeat the same feedback multiple times, it indicates Apollo is either not retaining feedback between iterations or producing partial fixes. This is the single biggest time sink in the revision process.
Task 2124: "build not loading" (×2) → "in 60s version there is an error: Cannot read properties of undefined" (×2) → "60s version redirects after 30s" (×2) → "60s version still ends before" (×2) — 8 messages, same root issue

How I Got To This & Resources I Had Access To

Data sources:

  • Production database: orchestration_states table — filtered for status = 'revision' in the last 7 days (March 25 – April 1, 2026)
  • S3 bucket apollo-tasks: retrieved all feedback-*/message.txt files for each task's chat history
  • Joined with orchestration, ideas, and games tables for full context
  • Date range: March 25, 2026 – April 1, 2026 (7 days)

Analysis approach:

  • Found 72 revision events in orchestration_states (all at "ideation" stage — these are PG-initiated revision requests)
  • Retrieved 207 actual feedback messages from S3 chat history across 65 unique tasks using concurrent S3 queries
  • Read all 207 messages and classified by issue type using keyword matching + manual review for ambiguous cases
  • Categories allow overlap (one feedback message can touch multiple issue types)

Important caveats:

  • All 72 revisions are at the "ideation" stage — this reflects the PG review flow, not auditor-to-Apollo feedback. These are customer-level or PG-level review requests
  • Feedback text is often brief/incomplete; some messages are follow-ups (e.g., "Fixed" or "Added") not actual issue descriptions
  • The repeat-message count inflates totals — some issues appear 4-8x for the same task before resolution
  • Some tasks from the 7-day window may have feedback messages outside the window if the task's original Apollo work started earlier
  • Classification is keyword-based with manual spot-checks; edge cases may be miscategorized
